<p>I mentioned <a href="https://techknowhow.library.emory.edu/content/xml-namespaces-and-email-headers">last week</a> that I was looking for an XML format for serializing email headers. I reached out to one of our metadata librarians, and she suggested taking a look at <a href="http://www.siarchives.si.edu/cerp/">CERP</a>, who recently spent a few years on a large research project developing just that. Overall it looks like a really good match: The XML schema that came from that research is aimed at taking large collections of email messaages and serializing them pretty directly into XML, from the account level all the way down to individual headers and message parts.</p>

<p>There&rsquo;s only one real mismatch between that project and the <a href="http://marbl.library.emory.edu/fc-salman-rushdie-papers.html">Rushdie project</a> that I&rsquo;m currently working on. The CERP XML schema roots each document in an <a href="http://www.records.ncdcr.gov/emailpreservation/mail-account/mail-account_docs.html#element_Account"><code>&lt;xm:Account&gt;</code></a> element, and folders and messages live inside that. The way the schema is written, any document that is rooted in any other element is technically invalid. And that&rsquo;s a minor problem for this project, though we can certainly work around it without much difficulty.</p>

<p>It&rsquo;s a problem because other aspects of the project lead us to model our data primarily at the message level. In particular, when we use <a href="http://lucene.apache.org/solr/">Apache Solr</a> to search for content, we want it to return individual email documents rather than a whole account document. That means we need to feed Solr one document per email. And <a href="http://defxws2006.cvt.dk/fedoragsearch/">Fedora GSearch</a> seems to have a strong preference for maintaining a one-to-one mapping between Solr documents and <a href="http://www.fedora-commons.org/">Fedora</a> objects, which means that we need to have one Fedora object per email message if we want GSearch to feed Solr correctly.</p>

<p>Fortunately, the CERP XML document doesn&rsquo;t technically say that we can&rsquo;t use a <a href="http://www.records.ncdcr.gov/emailpreservation/mail-account/mail-account_docs.html#type_message-type"><code>&lt;xm:Message&gt;</code></a> element outside of an <code>&lt;xm:Account&gt;</code>. In particular, it doesn&rsquo;t say we can&rsquo;t use one inside a <a href="http://fedora-commons.org/confluence/display/FCR30/Introduction+to+FOXML">FOXML</a> document. It just doesn&rsquo;t tell us how to do those, and it doesn&rsquo;t tell us how to validate them. In the grand scheme of things this isn&rsquo;t a huge problem. If we need to validate the CERP-in-FOXML documents (and I&rsquo;m not certain we&rsquo;ll ever need to), we&rsquo;ll need to write a schema that describes how to tie the two together. And if we ever want to provide someone a valid CERP document (which we probably will want), we&rsquo;ll need to store enough metadata that we can reproduce a <code>&lt;xm:Account&gt;</code> object. This is all easy.</p>

<p>So the upshot of all this is that I&rsquo;m using CERP&rsquo;s <a href="http://www.archives.ncdcr.gov/mail-account">Email XML serialization</a> to reposit email for <a href="http://marbl.library.emory.edu/">MARBL</a>&rsquo;s <a href="http://marbl.library.emory.edu/fc-salman-rushdie-papers.html">Rushdie</a> project, and I feel comfortable recommending it for anyone that needs to serialize RFC <a href="http://tools.ietf.org/html/rfc822" style="text-decoration:line-through">822</a> <a href="http://tools.ietf.org/html/rfc2822" style="text-decoration:line-through">2822</a> <a href="http://tools.ietf.org/html/rfc5322">5322</a> email as XML for long-term archival purposes. Good job, guys, and thanks.</p>
