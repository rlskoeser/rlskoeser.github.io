<p>A few quirks I ran into with South Migrations I thought it might be useful to have a headsup on.</p><p><em><strong>Problems with models and default value variables</strong></em></p><p>If you reference a default value from a variable then South will take the variable name as the default value instead of the value it references.&nbsp; So your initial startmigrate will run fine but subsequent startmigrate commands will throw errors. </p><p>For instance, I have a variable defining some choices on a field in a model and the line in the model looks like this:</p><p class="tt">PS_CHOICES = (<br />&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; (u'P', u'Pending Review'),<br />&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; (u'N', u'Not Adopted'),<br />&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; (u'W', u'Refine more'),<br />&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; (u'A', u'Approved for Development'),<br />&nbsp;&nbsp;&nbsp; )</p><p class="tt">planning_status = models.CharField(max_length=1, choices=PS_CHOICES, default=PS_CHOICES[0])</p><p>subsequent startmigrations throw an error about the reference to PS_CHOICES[0]</p><p>The way around this is to define the default as the literal value.&nbsp; So the example above works fine if I do:</p><p class="tt">planning_status = models.CharField(max_length=1, choices=PS_CHOICES, default='P')</p><p><em><strong>Problems with South and SQLite</strong></em></p><p>Be aware that SQLite does not play well with South Migrates as it doesn't support commands like 'alter_column' and such.&nbsp; Probably best to avoid south while you're working in SQLite.</p>
