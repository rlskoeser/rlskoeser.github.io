<p>I had a heck of a time finding documentation about <code>CModel</code> objects in the <a href="http://fedora-commons.org/">Fedora Commons</a> 3.0 <a href="http://fedora-commons.org/confluence/display/FCR30/CMA+Construction+Guide">Content Model Architecture</a>. Oh sure, they&rsquo;ve got documents describing <a href="http://fedora-commons.org/confluence/display/FCR30/Creating+a+Service+Definition">Service Definition</a> objects and <a href="http://fedora-commons.org/confluence/display/FCR30/Creating+a+Service+Deployment">Service Deployments</a>, but the actual Content Model objects were eluding me. Finally, through <a href="http://www.fedora-commons.org/confluence/display/DEV/Sample+Content+Model">examples</a> and some expertise from our local Fedora Repository Guru, <a href="https://techknowhow.library.emory.edu/blog/7">Rebecca</a>, I figured out a recipe that works.</p>

<p>At its simplest a <code>CModel</code> object&rsquo;s <code>RELS-EXT</code> stream must contain a <code>&lt;info:fedora/fedora-system:def/model#hasModel&gt;</code> property pointing to <code>rdf:resource</code> <code>&lt;info:fedora/fedora-system:ContentModel-3.0&gt;</code>. In other words, it has to say it&rsquo;s a <code>CModel</code> object. That usually involves putting something like this in the <code>RELS-EXT</code>:</p>
<pre>&lt;rdf:RDF xmlns:rdf="http://www.w3.org/1999/02/22-rdf-syntax-ns#"<br />         xmlns:cmodel="info:fedora/fedora-system:def/model#"&gt;<br />    &lt;rdf:Description rdf:about="info:fedora/mypidspace:myobjectpid"&gt;<br />        &lt;cmodel:hasModel rdf:resource="info:fedora/fedora-system:ContentModel-3.0"&gt;<br />    &lt;/rdf:Description&gt;<br />&lt;/rdf:RDF&gt;<br /></pre>

<p>In addition, the <code>CModel</code> object needs a <code>DS-COMPOSITE-MODEL</code> datastream. That datastream should have <code>CONTROL_GROUP="X"</code> and its <code>xmlContent</code> should validate against <a href="http://www.fedora-commons.org/definitions/1/0/dsCompositeModel.xsd">the dsCompositeModel XML Schema Document</a>. In English (if you&rsquo;re not familiar with XSD), that means it should have a <code>dsCompositeModel</code> element at the top level. That contains zero or mode <code>dsTypeModel</code> elements, each one representing a datastream guaranteed to be present on each object of that type. Each <code>dsTypeModel</code> element contains zero or more <code>form</code> elements describing acceptable content for that datastream. The <code>form</code> may have <code>MIME</code> and/or <code>FORMAT_URI</code> attributes constraining the datastream content type or format URI further. So the <code>DS-COMPOSITE-MODEL</code> for a <code>CModel</code> that requires an XML <code>FOO</code> datastream and a <code>image/png</code> <code>BAR</code> datastream looks something like this:</p>
<pre>&lt;dsCompositeModel xmlns="info:fedora/fedora-system:def/dsCompositeModel#"&gt;<br />  &lt;dsTypeModel ID="FOO"&gt;<br />    &lt;form MIME="application/xml"/&gt;<br />  &lt;/dsTypeModel&gt;<br />  &lt;dsTypeModel ID="BAR"&gt;<br />    &lt;form MIME="image/png"/&gt;<br />  &lt;/dsTypeModel&gt;<br />&lt;/code&gt;<br /></pre>

<p>Once you&rsquo;ve got that, you&rsquo;re ready to start defining <a href="http://fedora-commons.org/confluence/display/FCR30/Creating+a+Service+Definition"><code>SDef</code></a>s and <a href="http://fedora-commons.org/confluence/display/FCR30/Creating+a+Service+Deployment"><code>SDep</code></a>s for your new <code>CModel</code>.</p>
