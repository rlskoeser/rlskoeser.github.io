<p>The LSDI ingest script ran yesterday and successfully ingested 1,296 volumes into our test Fedora 3.2 instance (generating 936 Book objects and 1,296 Volume objects), with ingest errors only on 5 records.&nbsp; These records were then successfully harvested (via OAI-PMH, as MARCXML) into the Primo staging instance-- 936 records were pulled in, with no errors.</p><p>The ingest script ran for about 9 hours and ingested about 48GB of data (the majority of this number is made up by the PDF files, since the rest of the data is xml).</p><p>More technical details follow.</p><p>Time data on the ingest script:</p><pre>real&nbsp;&nbsp;&nbsp; 550m58.440s<br />user&nbsp;&nbsp;&nbsp; 116m50.393s<br />sys&nbsp;&nbsp;&nbsp;&nbsp; 15m43.882s</pre><p>Summary lines from the end of the log file:</p><pre>2009-06-19T00:52:13-04:00 NOTICE : Successfully ingested 1296 records<br />2009-06-19T00:52:13-04:00 NOTICE : Error ingesting 5 records</pre><p>Those 5 failures were all reasonable based on the test data provided to the ingest script from the Digitization Workflow (invalid paths to PDF files, missing path to a MARCXML file, and one record that did not have an indication that it had passed the public domain check).</p><p>The ingest script is pulling records marked as "Ready for Repository" from the latest development version of the digitization workflow, which has a recent copy of the production data, and a trimmed down copy of the filesystem with the relevant files needed for ingest (PDF &amp; MARCXML).&nbsp; Thanks to Jason for setting up this test instance of the Digitization Workflow and running the necessary scripts, and thanks to Chris Roddy and Steve for setting up a copy of the portion of the production filesystem that was needed for ingest.</p><p>This time, the ingest script ran against the pid soap service of the new django PID manager which is currently deployed for testing.</p><p>This is the second time we've run the LSDI ingest script in our test environment (there was a small problem in the xml that caused an error when attempting to load into PRIMO, which required some changes and re-ingesting the materials).&nbsp; For comparison, here are some numbers on the first time the ingest was attempted.</p><p>First ingest run on 2009-06-10:</p><pre>real&nbsp;&nbsp;&nbsp; 259m2.658s<br />user&nbsp;&nbsp;&nbsp; 50m50.154s<br />sys&nbsp;&nbsp;&nbsp;&nbsp; 8m18.562s</pre><p>Summary lines from the end of the ingest log file:</p><pre>2009-06-10T20:07:52-04:00 NOTICE : Successfully ingested 748 records<br />2009-06-10T20:07:52-04:00 NOTICE : Error ingesting 705 records</pre><p>In this case, there were errors with so many records for two main reasons: </p><ul><li>130 records failed to ingest because the collection name for the Theology Reference collection in the Digitization Workflow did not match what was in the LSDI ingest configuration</li><li>569 records failed to ingest because they had no indication of the public domain check passing, which Jason and I determined was a result of the task that runs the public domain check not completing on all of the records when I thought it had</li></ul><p>Jason helped me clean up the data on the records that were not ingested and I re-ran the ingest process again:</p><p>Second ingest run on 2009-06-11:</p><pre>real&nbsp;&nbsp;&nbsp; 262m51.408s<br />user&nbsp;&nbsp;&nbsp; 61m50.623s<br />sys&nbsp;&nbsp;&nbsp;&nbsp; 7m3.284s</pre><p>Summary lines from the end of the ingest log file:</p><pre>2009-06-11T23:31:20-04:00 NOTICE : Successfully ingested 509 records<br />2009-06-11T23:31:20-04:00 NOTICE : Error ingesting 196 records</pre><p>These ingest errors were all valid according to the data provided to the ingest script; the records that failed were, for the most part, records that were in the 'Ready for Repository' step in the Digitization Workflow, but that failed the public domain check.</p><p>&nbsp;</p><p>&nbsp;</p><p>&nbsp;</p>
