<p><a href="http://www.djangoproject.com/">Django</a> puts most of its runtime settings into a <a href="http://docs.djangoproject.com/en/1.0/topics/settings/">settings file</a>, but it creates some new complications along with the ones it solves. The settings file gets everything from project fundamentals like the <a href="http://docs.djangoproject.com/en/1.0/ref/settings/#installed-apps">installed applications</a> and <a href="http://docs.djangoproject.com/en/1.0/ref/settings/#middleware-classes">installed middleware classes</a> to private installation-specific settings like the <a href="http://docs.djangoproject.com/en/1.0/ref/settings/#secret-key">secret key</a> and <a href="http://docs.djangoproject.com/en/1.0/ref/settings/#database-password">database credentials</a>. The former ought to be in source control. The latter absolutely should not.</p>
<p>Lots of people have found <a href="http://kencochrane.blogspot.com/2009/01/managing-django-settings-across.html">interesting</a> <a href="http://martinjansen.com/2008/10/20/django-settings-files-for-development-and-production/">ways</a> to <a href="http://morethanseven.net/2009/02/15/let-you-sys-admin-override-your-django-settings/">solve</a> this <a href="http://blog.cottee.org/2008/04/django-settings-file.html">problem</a>, checking in the bits that should be checked in, while leaving the private bits private. Django gives us the <a href="http://docs.djangoproject.com/en/dev/topics/settings/#designating-the-settings">DJANGO_SETTINGS_MODULE</a> environment variable to pick which settings to use, but I&rsquo;ve found its actual use rather clunky in practice. Instead, here&rsquo;s what I use for personal projects and professional ones at <a href="http://web.library.emory.edu/">Emory University Libraries</a>:</p>
<p>As soon as you create a new Django project, separate local settings from <code>settings.py</code> into a new <code>localsettings.py</code>. Local settings include obviously private things like database credentials and the SECRET_KEY. It also includes things that probably vary between production and development environments, like <a href="http://docs.djangoproject.com/en/dev/ref/settings/#template-dirs">TEMPLATE_DIRS</a>, <a href="http://docs.djangoproject.com/en/dev/ref/settings/#database-name">DATABASE_NAME</a>, and <a href="http://docs.djangoproject.com/en/dev/ref/settings/#debug">DEBUG</a>. Leave these settings intact in <code>localsettings.py</code>, but remove them entirely from <code>settings.py</code>. At the end of <code>settings.py</code> add code akin to this:</p>
<pre>try:
    from localsettings import *
except ImportError:
    import sys
    print &gt;&gt;sys.stderr, 'No local settings. Trying to start, but if ' + \
        'stuff blows up, try configuring a localsettings.py for your system.'
    pass</pre>
<p>Since <code>localsettings.py</code> contains private information like database credentials, it should never be committed to source code control. Add <code>localsettings.py</code> to <a href="http://svnbook.red-bean.com/en/1.1/ch07s02.html#svn-ch-7-sect-2.3.3"><code>svn:ignore</code></a>, <a href="http://www.selenic.com/mercurial/hgignore.5.html"><code>.hgignore</code></a> or <a href="http://www.kernel.org/pub/software/scm/git/docs/gitignore.html"><code>.gitignore</code></a>, as appropriate for your project.</p>
<p>Of course, you should give administrators and other developers some idea of what belongs in the file, so copy <code>localsettings.py</code> to something obvious like <code>localsettings-sample.py</code>. Clean sensitive data out of that sample file and then add that to your repository. Now developers and production admins have a template that they can customize for per-installation settings by simply making their own <code>localsettings.py</code> from <code>localsettings-sample.py</code>, and there&rsquo;s less risk of someone accidentally committing a production password to source control.</p>
